# Task 1.11: main.zig — Minimal CLI

## Zig Tutorial — main.zig で初登場する文法

### T.1 `std.process.args()` — コマンドライン引数イテレータ

```zig
var args = std.process.args();
_ = args.skip(); // プログラム名をスキップ

while (args.next()) |arg| {
    // arg: []const u8 — 各引数の文字列
}
```

`std.process.args()` はイテレータを返す。C の `argv` に相当するが、
`argc` を持たずイテレータパターンで消費する。

`.skip()` は「次の要素を 1 つ読み飛ばす」。戻り値は `bool`
（要素があれば true、なければ false）だが、`_` で捨てている。
プログラム名（`argv[0]`）は通常不要なので最初にスキップ。

**`while` + Optional キャプチャ**:
`args.next()` は `?[]const u8` を返す。`while (expr) |val|` は
「expr が null でない間ループし、値を val にキャプチャする」。
01_07 の `if (optional) |val|` と同じキャプチャ構文を while で使う形。

### T.2 Zig 0.15 の I/O: `Io.Writer` vs `std.io.AnyWriter`

```zig
var stdout_buffer: [4096]u8 = undefined;
var stdout_writer = std.fs.File.stdout().writer(&stdout_buffer);
const stdout = &stdout_writer.interface;
```

Zig 0.15 で I/O システムが大幅に刷新された。2 つの Writer 型が共存する:

| 型                 | 位置               | バッファ | 状態       |
|-------------------|--------------------|---------|-----------|
| `Io.Writer`       | `std.Io.Writer`    | 内蔵     | **新 API** |
| `AnyWriter`       | `std.io.AnyWriter` | なし     | **非推奨** |

**新 `Io.Writer`**: バッファ付き Writer。`stdout.print(fmt, args)` で
バッファに書き込み、`stdout.flush()` で実際に出力する。
`flush()` を忘れると出力が画面に表示されない。

**旧 `AnyWriter`**: vtable ベースの型消去 Writer（01_08 で使用）。
`Io.Writer` とは互換性がない。

**なぜ 2 つあるか**: Zig 0.15 は I/O の移行期。`AnyWriter` は非推奨だが、
`std.fmt.format(writer, ...)` のような既存 API がまだ `AnyWriter` を
要求する場合がある。form.zig の `formatPrStr` も `AnyWriter` を受け取る。

**本タスクでの回避策**: `formatPrStr` を直接 stdout に使えないため、
一旦 `toString()` で文字列に変換してから `stdout.print("{s}", ...)` で出力。

```zig
const s = form.?.toString(alloc) catch |err| { ... };
try stdout.print("{s}\n", .{s});
```

### T.3 `std.process.exit(n)` — 即座のプロセス終了

```zig
std.process.exit(1);
```

C の `exit(1)` に相当。ステータスコード n で即座にプロセスを終了する。
`defer` や `errdefer` は**実行されない**（OS がプロセスごと回収する）。

Zig の `main` は `!void` を返すので、通常は `return error.X` でエラー終了する。
しかし CLI ツールではユーザ向けメッセージを出してから `exit` する方が親切:

```zig
// エラーメッセージ → exit(1) パターン
try stderr.print("Error: -e requires an argument\n", .{});
try stderr.flush();
std.process.exit(1);

// vs. エラー返却パターン（Zig がスタックトレースを出す → ユーザには不親切）
return error.MissingArgument;
```

### T.4 `@errorName(err)` — エラー値の文字列取得

```zig
reader.read() catch |err| {
    try stderr.print("Read error: {s}\n", .{@errorName(err)});
    //                                       ^^^^^^^^^^
    //                  error.SyntaxError → "SyntaxError" ([]const u8)
```

`@errorName(err)` はエラー値を `[]const u8` 文字列に変換するビルトイン。
コンパイル時にエラー名の文字列テーブルが生成され、ランタイムでは
テーブルルックアップのみ。

**01_03 との対比**: error.zig では `setErrorFmt` でエラー詳細を
threadlocal に格納していた。`@errorName` はそのような詳細なしで
エラータグ名だけを返す。Phase 2 以降で error.zig のフォーマッタと
組み合わせてより詳細なエラーメッセージを実装する。

### T.5 `catch |err| { ... }` — エラーキャプチャ付きブロック

```zig
const form = reader.read() catch |err| {
    try stderr.print("Read error: {s}\n", .{@errorName(err)});
    try stderr.flush();
    std.process.exit(1);
};
```

01_03 では `catch blk: { break :blk default_value; }` を学んだ。
ここでは `catch |err| { ... }` でエラー値をキャプチャし、
メッセージ表示後に `exit(1)` で終了する。

**`catch` ブロックの戻り値**: 通常は `catch` ブロックの結果が式の値になる。
しかし `std.process.exit` は `noreturn` 型なので、ブロック全体も
`noreturn` として扱われ、`form` の型は `?Form`（error union が取り除かれた型）になる。

### T.6 `form.?` — Optional の強制展開

```zig
const s = form.?.toString(alloc) catch |err| { ... };
```

`form` は `?Form` 型（`reader.read()` が `ReadError!?Form` を返し、
`catch` で error を処理した後の残り）。`.?` は null でないことを前提に
値を取り出す。null なら unreachable パニック。

01_07 では `if (optional) |val|` で安全に展開する方法を学んだ。
`.?` はその短縮形で、「ここは絶対に null でない」と分かっている場合に使う。
このコードでは `if (form == null) break;` で null チェック済みなので安全。

## 処理フロー

```
main()
  │
  ├─ 引数なし → "ClojureWasm" を表示
  │
  ├─ -e <expr> → read + print ループ
  │    │
  │    ├─ ArenaAllocator 確保
  │    ├─ Reader.init(alloc, source)
  │    └─ while: reader.read() → form.toString() → stdout.print()
  │
  └─ -h / --help → ヘルプ表示
```

## CLI 引数パース

```zig
while (args.next()) |arg| {
    if (std.mem.eql(u8, arg, "-e") or std.mem.eql(u8, arg, "--eval")) {
        expr = args.next();           // 次の引数が式
        if (expr == null) { ... }     // なければエラー
    } else if (std.mem.eql(u8, arg, "--help") or std.mem.eql(u8, arg, "-h")) {
        // ヘルプ表示
    } else {
        // 不明なオプション
    }
}
```

`std.mem.eql(u8, arg, "-e")` は文字列比較。Zig にはビルトインの
文字列比較演算子（`==`）がないため、`std.mem.eql` を使う。

Phase 14 で本格的な CLI（ファイル読み込み、REPL、deps.edn）に拡張する。
現段階では `-e` と `-h` のみ。

## メモリ管理

```zig
var arena = std.heap.ArenaAllocator.init(std.heap.page_allocator);
defer arena.deinit();
const alloc = arena.allocator();
```

ArenaAllocator を使い、プロセス終了時に `defer` で一括解放。
短命な CLI 実行では個別 free のオーバーヘッドを避ける。
Reader が内部でコレクションの `[]Form` やエスケープ処理済み文字列を
確保するので、アロケータは必須。

`toString()` も内部で `ArrayList(u8)` を使うため、同じ arena から確保される。
arena.deinit() ですべて一括解放されるため、個別の free は不要。

## Form → 文字列の経路

```zig
// form.zig
pub fn toString(self: Form, alloc: std.mem.Allocator) ![]u8 {
    var buf: std.ArrayList(u8) = .empty;
    try self.formatPrStr(buf.writer(alloc).any());  // AnyWriter 経由
    return buf.toOwnedSlice(alloc);
}
```

`formatPrStr` は `std.io.AnyWriter` を受け取る（01_08 で解説）。
`ArrayList.writer(alloc).any()` で AnyWriter に変換する。
Zig 0.15 の新 `Io.Writer` とは互換性がないため、
stdout への直接ストリーミングではなく `toString` → `print` の 2 段階。
