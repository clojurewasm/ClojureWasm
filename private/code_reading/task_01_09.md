# Task 1.9: eval/tokenizer.zig — Lexer

## Zig Tutorial — tokenizer.zig で初登場する文法

### T.1 `switch` on `u8` — バイト値による分岐

```zig
const ch = self.source[self.pos];
switch (ch) {
    '(' => { self.advance(); return .{ .kind = .lparen, ... }; },
    ')' => { self.advance(); return .{ .kind = .rparen, ... }; },
    '"' => return self.readString(start, start_line, start_col),
    ':' => return self.readKeyword(start, start_line, start_col),
    else => { ... },
}
```

01_02 の `switch (self.tag())` は `enum` 値による分岐だったが、
ここは `u8` （バイト値）による分岐。`'('` は Zig の文字リテラルで `u8` 型。

Zig の `switch` は enum でも整数でもパターンマッチに使える。
`else =>` は「上で列挙されなかったすべての値」をキャッチする。

**文字リテラル `'\n'`, `'\t'`, `'\x0C'`**:

```zig
fn isWhitespace(c: u8) bool {
    return switch (c) {
        ' ', '\t', '\n', '\r', '\x0C', ',' => true,
        //                      ^^^^^^ フォームフィード (0x0C)
        else => false,
    };
}
```

`'\x0C'` は 16 進エスケープ。`\x` + 2 桁の hex で任意のバイト値を表す。
`\n` (0x0A) や `\t` (0x09) は名前付きエスケープ。

### T.2 ステートフルイテレータパターン

```zig
pub const Tokenizer = struct {
    source: []const u8,    // 入力全体（不変）
    pos: u32 = 0,          // 現在位置
    line: u32 = 1,         // 現在行
    column: u16 = 0,       // 現在列

    pub fn next(self: *Tokenizer) Token { ... }
};
```

`next(self: *Tokenizer)` は `self` を変更する（`pos`, `line`, `column` を進める）。
呼び出し側は `var t = Tokenizer.init(src); const tok = t.next();` と使う。

**`*Tokenizer` (mutable pointer)**:
01_05 で出た `*Cons` と同じ。`self: Tokenizer` だとコピーされて
内部状態の変更が呼び出し側に反映されない。`self: *Tokenizer` で
「呼び出し側の変数を直接変更する」ことを保証する。

**`const` vs `var` の使い分け**:
`Tokenizer.init(src)` の結果を `var t` に入れる必要がある。
`const t` にすると `t.next()` で状態を変更できない（コンパイルエラー）。
Zig は「変更するなら var、しないなら const」を言語レベルで強制する。

### T.3 `@intCast` — 安全な整数縮小

```zig
fn makeToken(self: *Tokenizer, kind: TokenKind, start: u32, ...) Token {
    const len = self.pos - start;   // u32
    return .{ ..., .len = @intCast(len), ... };
    //                    ^^^^^^^^^ u32 → u16 に縮小
}
```

`Token.len` は `u16` だが、`self.pos - start` は `u32`。
`@intCast` は「値が収まるなら変換、収まらないなら Debug ビルドでパニック」。

01_06 で出た `@truncate` との違い:

| ビルトイン    | オーバーフロー時 | 用途                     |
|-------------|----------------|--------------------------|
| `@intCast`  | パニック (安全)  | 値が収まると期待する場合   |
| `@truncate` | 上位ビット切り捨て | ハッシュ等で意図的に切り捨て |

Token の長さは 65535 を超えないと期待する → `@intCast` が適切。
Murmur3 の `usize → u32` は意図的な切り捨て → `@truncate` が適切。

### T.4 モジュールレベルの private 関数

```zig
fn isDigit(c: u8) bool {
    return c >= '0' and c <= '9';
}

fn isWhitespace(c: u8) bool {
    return switch (c) { ... };
}
```

`pub` なしの `fn` はファイル内 (モジュール内) でのみアクセス可能。
01_03 の `setErrorFmt` などは `pub fn` だったが、文字分類関数は
外部から使う理由がないので private。

Zig のアクセス制御は 2 段階のみ: `pub`（外部公開）と無印（ファイル内）。
Java の `protected` や C++ の `friend` に相当するものはない。
シンプルだが、モジュール（= ファイル）を小さく保つことが前提。

### T.5 `and` / `or` の短絡評価

```zig
if (self.pos + 1 < self.source.len and isDigit(self.source[self.pos + 1])) {
```

Zig の `and` は短絡評価 (short-circuit evaluation)。
左辺が `false` なら右辺を評価しない。C の `&&` と同じ動作。

これにより、`self.pos + 1 < self.source.len` が false のとき
`self.source[self.pos + 1]` にアクセスしない（範囲外アクセス回避）。

Zig は配列の範囲外アクセスで安全性パニックを起こすため、
この短絡評価パターンは頻繁に使われる。
C では同じコードが buffer overread になりうるが、Zig では
Debug/ReleaseSafe ビルドでランタイムチェックが入る。

## 概要

Tokenizer は **テキスト → トークン列** の変換を担当する。
Reader パイプラインの第 1 段階:

```
Source text → [Tokenizer] → Token stream → [Reader] → Form tree
```

トークンは**ソースバッファへの参照**（start + len）であり、コピーしない。
ゼロコピー設計により、100KB のソースを読んでも追加メモリは Token 構造体分だけ。

## Token 構造体

```zig
pub const Token = struct {
    kind: TokenKind,      // 1 byte — トークン種別
    start: u32,           // 4 bytes — ソース内のバイトオフセット
    len: u16,             // 2 bytes — トークンの長さ
    line: u32,            // 4 bytes — 1-based 行番号
    column: u16,          // 2 bytes — 0-based 列番号
};
```

**`text(source)` メソッド**: トークンのテキストを取得するとき、
Token 自体は文字列を持たず、`source[start..start+len]` でスライスを返す。

```zig
pub fn text(self: Token, source: []const u8) []const u8 {
    const s: usize = self.start;
    const e: usize = s + self.len;
    return source[s..e];
}
```

`u32` → `usize` の変換が必要（スライスインデックスは `usize`）。
Zig では異なる整数型の直接比較・演算はコンパイルエラーになるため、
ローカル変数で明示的に変換。`u32` → `usize` は拡大変換なので安全
（暗黙変換はされないが、Zig は型の混在を許さない設計）。

## 文字分類

Clojure の文字分類は通常の言語と異なる点がある:

```zig
fn isWhitespace(c: u8) bool {
    return switch (c) {
        ' ', '\t', '\n', '\r', '\x0C', ',' => true,  // ←カンマ！
        else => false,
    };
}
```

**カンマはホワイトスペース**: Clojure では `,` は完全に無視される。
`{:a 1, :b 2}` と `{:a 1 :b 2}` は同義。これにより map の可読性が向上する。

**ターミネータ**: トークンの区切りとなる文字:

```zig
fn isTerminator(c: u8) bool {
    return isWhitespace(c) or switch (c) {
        '"', ';', '(', ')', '[', ']', '{', '}', '\\', '#' => true,
        else => false,
    };
}
```

`isSymbolChar` は `!isTerminator(c) and c > ' '` で定義。
つまり「ターミネータでも制御文字でもない任意の文字」がシンボル構成文字。
これにより `+`, `-`, `*`, `/`, `!`, `?`, `<`, `>` 等がシンボルに使える。

## 数値パース

数値の判定はトークナイザレベルでは簡易的:

```
整数:    42, -7, +3, 0xFF
浮動小数: 3.14, 1e10, 2.5E-3
```

**符号付き数値 vs シンボル**: `+` と `-` は単独ならシンボルだが、
直後に数字が続けば数値の符号。判定ロジック:

```zig
if ((ch == '+' or ch == '-') and self.pos + 1 < self.source.len and isDigit(self.source[self.pos + 1])) {
    return self.readNumber(...);
}
```

この 3 条件の `and` はすべて短絡評価される。左から順に:
1. `ch == '+'` or `ch == '-'` — 符号文字か？
2. `self.pos + 1 < self.source.len` — 次の文字が存在するか？
3. `isDigit(self.source[self.pos + 1])` — 次の文字が数字か？

2 が false なら 3 は評価されない（範囲外アクセス回避）。

**N / M サフィックス**: `42N` (BigInt) と `3.14M` (BigDecimal) は
Phase 1 では実質サポートしないが、トークナイザレベルでは受理してスキップする。
Reader/Analyzer 段階でエラーにするか、将来の Phase で対応する。

## ソース位置追跡

`advance()` が 1 文字進むたびに line/column を更新:

```zig
fn advance(self: *Tokenizer) void {
    if (self.pos < self.source.len) {
        if (self.source[self.pos] == '\n') {
            self.line += 1;
            self.column = 0;
        } else {
            self.column += 1;
        }
        self.pos += 1;
    }
}
```

**重要な点**: `advance()` は「現在の `pos` の文字」を見て line/column を更新し、
その後 `pos` を進める。つまり改行 `\n` を読んだとき:
1. `\n` を検出 → line をインクリメント、column を 0 にリセット
2. pos を進める → 次の文字は新しい行の先頭

改行の「次」の文字は column=0 から始まる。
`advance` がもう一度呼ばれると column が 1 になる。

実際のテストで確認:
```zig
test "source location tracking" {
    var t = Tokenizer.init("(\n  42)");
    // '(' → line=1, col=0  (先頭文字)
    // '4' → line=2, col=2  (改行後2スペースのインデント)
    // ')' → line=2, col=4
}
```

## コメントとシバン

**行コメント**: `;` 以降は行末まで skipWhitespace 内でスキップ:

```zig
} else if (c == ';') {
    while (self.pos < self.source.len and self.source[self.pos] != '\n') {
        self.advance();
    }
}
```

コメントを skipWhitespace のループ内で処理するため、再帰は不要。
ホワイトスペースとコメントが交互に来ても正しくスキップされる。

**シバン**: `#!` はディスパッチマクロとして処理し、行末まで読み飛ばしてから
`self.next()` を再帰呼び出しして次のトークンを返す。

```zig
'!' => {
    while (self.pos < self.source.len and self.source[self.pos] != '\n') {
        self.advance();
    }
    return self.next(); // ← 再帰で次のトークンを返す
},
```

深い再帰にはならない（shebang はファイル先頭 1 行のみ）。
コメントの skipWhitespace とは設計が異なる理由: `#!` は `readDispatch` 内で
処理され、ここから skipWhitespace に戻るより再帰の方がシンプル。
