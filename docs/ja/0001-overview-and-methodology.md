# 0001 — はじめに / 学習方法論

> 所要時間: 30〜45 分（読むだけ）

ClojureWasm を **Zig 0.16 でゼロから作る** 過程を、概念単位の章で
追体験するための教科書。各章は本リポジトリ
[`ClojureWasmFromScratch`](../../) の git 履歴上の特定のコミット
（あるいは複数）の **設計判断** と **コード実装** を、教科書として
読める形に圧縮したものです。

---

## この章で学ぶこと

- この教科書の **想定読者** と **読み方**
- 各章が採用している **5 つの学習テクニック**（再構成法 / 予測検証 /
  L1L2L3 / Feynman / マイクロカタ）
- **`git checkout` によるタイムマシン操作**で歴史を追体験する手順
- 章のフロントマター（`commits:` / `related-tasks:` / `related-chapters:`）
  の意味と使い方
- なぜ **概念単位** で章立てし、**コミット単位ではない** のか

---

## 1. 想定読者と前提知識

### この教科書はだれのため？

主たる読者は 3 種類想定されています：

1. **未来の自分**: 半年後にコードを読み返したとき「なぜそうしたか」を
   思い出すための補助線。git log だけでは復元できない設計判断を保存。
2. **Conj 2026 / 技術書の読者**: ClojureWasm を「コードベースが
   読める Clojure ランタイム」として学びたい後発者。
3. **Clojure Conj の発表前後で議論したい同業者**: なぜ JVM なし、
   なぜ Zig、なぜ NaN boxing Day 1 から、を実装と一緒に説明できる。

### 前提知識

| 領域 | 期待値 |
|------|--------|
| Clojure | `(defn f [x] (+ x 1))` が読めればよい。コアライブラリの細部は不要。 |
| Zig | 0.16 のドキュメント (release notes) を一度通読していること。本教材で 0.16 イディオムは都度説明する。 |
| 処理系一般 | tagged union と再帰下降パーサが何かわかる。 |
| 数値表現 | IEEE-754 浮動小数点と 16 進数（NaN boxing の前提）。第 0002 章で復習。 |

すべて未経験という方は、まず `~/Documents/MyProducts/learn-clojurewasm-
from-scratch/code_learning/00_methodology.md` を読むことを推奨。

## 2. 学習テクニック（5 種類）

### 2.1 再構成法 (Reconstruct from Memory)

最も効果が高い。手順:

1. 章の 1 つの節を読む
2. **章を閉じる**
3. 白紙（or 新しいファイル）に、読んだ内容をゼロから書く
4. 詰まったら「何が思い出せないか」をメモしてから戻る

**ポイント**: 「思い出せない」瞬間こそ学習が起きている。すぐ見返す
のはもったいない。30 秒は粘る。

### 2.2 予測 → 検証ループ (Predict-Then-Verify)

1. コードの出力（または振る舞い）を **紙に書いて予測する**
2. `bash test/run_all.sh` または `cljw -e "..."` で検証する
3. 合っていたら理解 OK。間違っていたら原因を分析

これは "testing effect"（認知科学）。予測を外した体験は、正解を読む
だけより 2〜3 倍記憶に残ります。

### 2.3 段階的写経 (L1 / L2 / L3)

各章末の演習は次の 3 段で配置されています：

| Level | やること                              | 負荷 |
|-------|---------------------------------------|------|
| L1: 穴埋め    | コードの一部が空欄。正しい値を埋める   | 低   |
| L2: 部分再構成 | シグネチャだけ与え、本体を書く          | 中   |
| L3: 完全再構成 | ファイル名 / 公開 API のみ与え書き起こす | 高   |

**L1 で手が温まってから L3 に進むとスムーズ**。いきなり L3 を試すと
挫折しやすい。

### 2.4 Feynman Technique（説明課題）

「6 歳の自分に説明するつもりで書く」。各章末に 3 問あります。書けな
ければ理解が不完全と判定して、その節をもう一度読む。

### 2.5 マイクロカタ

1 つの概念に集中した 5〜15 分の小演習。本リポでは演習折り畳みの中で
採用。`zig test <ファイル>` で即座にフィードバックが得られます。

## 3. 章の構造

すべての章は `.claude/skills/code-learning-doc/TEMPLATE_PHASE_DOC.md`
の形式に従います：

```
---
chapter: NN
commits: [...]
related-tasks: [...]
related-chapters: [...]
---

# NN — タイトル

## この章で学ぶこと  (3-5 行)
## 1. 概念 A          ← 演習 N.1 (L1)
## 2. 概念 B          ← 演習 N.2 (L2)
## 3. 概念 C          ← 演習 N.3 (L3)
## 4. 設計判断と却下した代替
## 5. 確認 (Try it)
## 6. 教科書との対比 (v1 / v1_ref / Clojure JVM / Babashka)
## 7. Feynman 課題 (3 問)
## 8. チェックリスト  (5 項目)
## 次へ → NN+1
```

演習の **答えは `<details>` で折りたたまれて** います。マウスで開く
までは見えない。**手を動かす前に開かない**ことが学習の核心。

## 4. `git checkout` でタイムマシン

各章のフロントマターには `commits:` SHA リストがあります。これは
その章が描写する **本リポジトリの特定の時点** を指します。

```sh
# 例: 第 0002 章 (NaN boxing) の時点に時計を巻き戻す
cd ~/Documents/MyProducts/ClojureWasmFromScratch
git checkout 8b487f9    # NaN boxing が入った直後の状態

# その時点のコードを読む / テストを動かす
zig build test

# 戻る
git checkout cw-from-scratch
```

`git stash` を併用すると、自分の実験を保ったままタイムマシンできる：

```sh
git stash       # 自分の実験を退避
git checkout 8b487f9
# ... 観察 ...
git checkout cw-from-scratch
git stash pop   # 退避した実験を戻す
```

**この教科書のキモ**は、各章で **「今の HEAD は何が無くて、これから
何を加えるのか」** が常に明示されることです。**雪だるま式に増えて
いく実装** を、概念単位で追体験できる。

---

## 5. なぜ「概念単位」で章立てするのか

### 旧形式（廃止）の問題

以前は `1 Phase = 1 doc`（diary 形式）で書いていました。1 章 = 11
コミット = 約 36 行/コミット という密度になり、**教科書として浅く、
事件記録として冗長** という両方失格の状態でした。

そこで：

- **章 = 概念**（NaN boxing、Reader、Analyzer、TreeWalk、…）
- **章 ≠ コミット**（1 概念は 1〜5 個のコミットを背景に持つ）
- **章 = 教育単位**（演習・予測検証・Feynman・チェックリスト込み）

に切り替えました。

### コミットメモは `private/notes/` に分離

それでも「task 単位の事件メモ」自体には価値があります。詰まった
ポイント / 教科書との対比 / 章を書くときの伏線 / 却下案 など。
これらは **`private/notes/`（gitignored）の per-task note** として
**書いた直後、熱いうちに**残し、章を書くときの一次資料にします。

詳しくは `.claude/skills/code-learning-doc/SKILL.md` を参照。

## 6. 進め方の推奨ペース

| 進め方 | 1 日のペース | 復習 |
|--------|-------------|------|
| 速度重視 | 章 2 つ / 日、演習は L1 のみ | 翌日に L2 を埋め直し |
| 標準     | 章 1 つ / 日、L1 + L2 | 翌日に L3 を再構成 |
| 深堀り   | 章 1 つ / 2 日、L1〜L3 + Feynman | 1 週後に章タイトルだけ見て再構成 |

**復習は L3 再構成で**：1 日後に前章をファイル名と公開 API だけ
見て中身を書くと、驚くほど定着します。エビングハウスの忘却曲線に
抗う最も強い方法。

## 9. 演習 1.3: 自分の学習スタイルを設計 (L3)

「自分はこの教科書をどう進めるか」を **白紙に書き起こします**。

要求:

- 1 日のペースを決める（速度重視 / 標準 / 深堀り）
- 復習のタイミング（翌日 L2 / 翌日 L3 / 1 週後 タイトルのみ）
- 演習をどこまで解くか（L1 のみ / L1+L2 / L1+L2+L3）
- Feynman 課題に答える媒体（紙 / scratch.clj / Anki）
- 詰まったときの逃げ道（reference clones を grep する / Claude に
  訊く / 1 章戻る）

<details>
<summary>例</summary>

休日: 前週の章を 3 つピックアップ、ファイル名だけ見て L3 再構成

            grep してから章に戻る

これは例。自分の生活リズムと既知の知識量に合わせる。

## 10. 設計判断と却下した代替

| 案 | 採否 | 理由 |
|----|------|------|
| 章 = 概念 + 演習 | ✓ | 教育目的に合致。半年後の自分にも有用 |
| 章 = 1 コミット | ✗ | 36 行/章 では薄すぎ、冗長な diary になる |
| 章 = 1 Phase | ✗ | 11 コミットを雑に圧縮、読み手が概念で迷う |
| 演習なし | ✗ | testing effect が活用できず、定着が悪い |

ROADMAP §12.4 の「8-step TDD loop」と整合：Step 7 の per-task note が
本教材の一次資料、Step 6 の chapter は本ディレクトリへ着地。
