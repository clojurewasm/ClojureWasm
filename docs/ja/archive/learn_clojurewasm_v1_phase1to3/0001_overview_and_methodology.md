# 0001 — はじめに / 教科書としての方針

> 所要時間: 20〜30 分（読むだけ）

ClojureWasm を **Zig 0.16 でゼロから作る** 過程を、概念単位の章で
追体験するための教科書です。各章は本リポジトリ
[`ClojureWasmFromScratch`](../../) の git 履歴上の特定のコミット
（あるいは複数）の **設計判断** と **コード実装** を、教科書として
読める形に圧縮したものです。

本章は教科書全体の入り口として、想定読者・章の構造・コミットとの
対応・読み進め方を **読むだけで把握できる** ように整理しています。

---

## この章で学ぶこと

- 本教科書の **想定読者** と **前提知識**
- 各章が採用している **純粋解説** の方針と、その背景にある考え方
- 章のフロントマター（`commits:` / `related-tasks:` /
  `related-chapters:`）の意味
- **`git checkout` によるタイムマシン操作**で章とコードを行き来する手順
- なぜ **概念単位** で章立てし、**コミット単位ではない** のか

---

## 1. 想定読者と前提知識

想定読者は次の 3 種類です。

1. **未来の自分**: 半年後にコードを読み返したとき「なぜそうしたか」を
   思い出すための補助線。`git log` だけでは復元できない設計判断を
   ここに保存しておきます。
2. **Conj 2026 / 技術書の読者**: ClojureWasm を「コードベース全体を
   読み通せる Clojure ランタイム」として学びたい後発者。
3. **Clojure Conj 前後で議論したい他の Clojurian**: なぜ JVM なし、
   なぜ Zig、なぜ NaN boxing を Day 1 から、を実装と並べて説明
   できるようにする。

前提知識は次の通りです。深い Zig 知識は不要で、0.16 固有のイディオムは
本文で都度説明します。

| 領域       | 期待値                                                               |
|------------|----------------------------------------------------------------------|
| Clojure    | `(defn f [x] (+ x 1))` が読めればよい。コアライブラリの細部は不要    |
| Zig        | 0.16 のリリースノートに一度目を通している程度                        |
| 処理系一般 | tagged union と再帰下降パーサが何かわかる                            |
| 数値表現   | IEEE-754 浮動小数点と 16 進数（NaN boxing の前提）。第 0002 章で復習 |

## 2. 教科書としての方針 — 純粋解説

本教科書は **純粋な解説（pure exposition）** で書かれています。
読者は章を読み流すだけで概念を獲得できることを目指し、章末問題・
予測検証プロンプト・段階的写経・Feynman 課題・チェックリストといった
「能動的な演習」は **置きません**。

この方針は学習法の優劣を主張するものではなく、**この教科書という
媒体での役割分担** に基づきます。

- **教科書本体は概念の保存と読解の場** とする。地の文と code 抜粋の
  密度で勝負し、読み終えたとき「結局のところこの章は X だ」と
  口頭で再現できる状態を作る。
- **手を動かす学習** は実コード（`bash test/run_all.sh`、`zig build`、
  `cljw -e "..."`）と `git checkout` による履歴操作に委ねる。
  各章末の「確認 (Try it)」は読者を試すための章末問題ではなく、
  **その章の状態を手元で再現する最短手順** です。

したがって本教科書は：

- **1 章 = 1 概念群**。NaN boxing、Reader、Analyzer、TreeWalk のように、
  独立して理解できる単位で章を切ります。
- **1 セクション = 1 概念**。セクションだけ読んでも意味が通る粒度で
  書き、重要な数値や bit pattern は表で整理し、地の文では
  「なぜこの形か」を語ります。
- **逐次読み**を前提とします。章は前章の知識を踏まえて積み上がるので、
  原則として番号順に読んでください。

## 3. 章の構造（テンプレ）

すべての章は `.claude/skills/code_learning_doc/TEMPLATE_PHASE_DOC.md`
の形式に従います。

```
---
chapter: NN
commits: [...]
related-tasks: [...]
related-chapters: [...]
---

# NN — タイトル

## この章で学ぶこと          (3-5 行)
## 1. 概念 A                  ← 解説 + コード抜粋
## 2. 概念 B                  ← 解説 + コード抜粋
## 3. 概念 C                  ← 解説 + コード抜粋
## (必要に応じて概念を追加)
## N. 設計判断と却下した代替   (表)
## N+1. 確認 (Try it)          (実行可能スニペット)
## N+2. 教科書との対比         (v1 / v1_ref / Clojure JVM / Babashka)
## この章で学んだこと          (1〜3 行 / 1〜3 個の箇条書きで凝縮)
## 次へ → NN+1
```

「**この章で学ぶこと**」は章頭の学習目標、「**この章で学んだこと**」
は章末の総括で別物です。後者は読み終えた読者が口頭で 30 秒で再現
できる結論文を 1〜3 個に絞り、「結局のところこの章は X だ」と言い切る
形で書きます。

「**設計判断と却下した代替**」の表は、なぜ採用案を選び、なぜ他の案を
却下したかを 1 行ずつで残します。`git log` には残らない情報を、
読み手が後から復元できる形にしておく場所です。

「**教科書との対比**」では本リポジトリのほかに、参照クローンとして
v1（`~/Documents/MyProducts/ClojureWasm`）、v1_ref（前回試行）、
Clojure JVM、Babashka と並べ、本リポジトリがどこで分岐したかを
明記します。

## 4. `git checkout` でタイムマシンする

各章のフロントマターには `commits:` SHA リストがあります。これは
その章が描写する **本リポジトリの特定の時点** を指します。章とコードを
行き来する基本動作はこうです。

```sh
# 例: 第 0002 章 (NaN boxing) の時点に時計を巻き戻す
cd ~/Documents/MyProducts/ClojureWasmFromScratch
git checkout 8b487f9    # NaN boxing が入った直後の状態

# その時点のコードを読む / テストを動かす
zig build test

# 戻る
git checkout cw-from-scratch
```

自分の実験を抱えたままタイムマシンしたいときは `git stash` を併用
します。

```sh
git stash       # 自分の実験を退避
git checkout 8b487f9
# ... 観察 ...
git checkout cw-from-scratch
git stash pop   # 退避した実験を戻す
```

この仕組みによって、各章で **「今の HEAD には何が無く、これから何を
加えるのか」** が常に明示されます。雪だるま式に積み上がっていく
実装を、概念単位で追体験できます。

---

## 5. なぜ「概念単位」で章立てするのか

以前は `1 Phase = 1 doc`（diary 形式）で書いていました。1 章 = 11
コミット = 約 36 行 / コミット という密度で、**教科書としては浅く、
開発記録としては冗長** という両面で不十分でした。

そこで現在は次のように切り替えています。

- **章 = 概念**（NaN boxing、Reader、Analyzer、TreeWalk、…）
- **章 ≠ コミット**（1 概念は 1〜5 個のコミットを背景に持つ）
- **章 = 教育単位**（純粋解説、独立して読める、概念で並ぶ）

章を概念単位にしたことで、`git log` を時系列に追わなくても、
**読みたい概念だけを取り出して読める** ようになりました。逆に「task
単位の作業メモ」は章とは別の媒体に分離しました。詰まったポイント、
教科書との対比、却下案などの **per-task note** は
`private/notes/`（gitignored）に、書いた直後・記憶が熱いうちに残し、
章を書くときの一次資料として使います。詳しくは
`.claude/skills/code_learning_doc/SKILL.md` を参照してください。

---

## 6. 設計判断と却下した代替

| 案                                        | 採否 | 理由                                                                                   |
|-------------------------------------------|------|----------------------------------------------------------------------------------------|
| 章 = 概念、純粋解説                       | ✓   | 読み流しで概念が掴め、半年後の自分にも有用。独立した章として再利用しやすい             |
| 章 = 1 コミット                           | ✗   | 36 行 / 章では薄すぎて、冗長な diary になる                                            |
| 章 = 1 Phase                              | ✗   | 11 コミットを雑に圧縮することになり、読み手が概念で迷う                                |
| 演習駆動（L1/L2/L3 + Feynman + 章末問題） | ✗   | 教科書本体の役割を「概念の保存」と「読み通せる解説」に絞り、能動学習は実コードに委ねる |

ROADMAP §12.4 の「8-step TDD loop」と整合しています。Step 7 の
per-task note が本教材の一次資料となり、Phase 境界で章として
本ディレクトリへ着地します。

## 8. 教科書との対比

| 軸               | learn-cw（参考リポ）           | 本教科書                                    |
|------------------|--------------------------------|---------------------------------------------|
| 想定実装         | v1_ref（前回試行）             | 本リポジトリ（cw-from-scratch）             |
| 章数             | 19 + 19（learning + reading）  | ~16（統合）                                 |
| 形式             | 演習駆動（L1/L2/L3 + Feynman） | 純粋解説（読むことに集中、能動演習なし）    |
| Phase カバレッジ | Phase 1+2 のみ                 | Phase 1+2 から開始、以後拡張                |
| 配置             | 別リポジトリ                   | 同一リポジトリ `docs/ja/learn_clojurewasm/` |

本リポジトリの理念で整理した点：

- **教材を実装本体と同じ git 履歴に同居** させ、`git checkout` でいつ
  でも実コードに帰れるようにした。
- **演習駆動から純粋解説に切り替え**、教科書本体は概念の保存と読解に
  特化し、能動学習は実コード（テスト・REPL・履歴操作）に委ねる。

## この章で学んだこと

- 本教科書は **読むことに振り切った純粋解説** であり、章末問題は
  置かない。手を動かす学習は実コードと `git checkout` に委ねる。
- 章は **概念単位**（NaN boxing / Reader / Analyzer …）で切り、
  1 章 = 1 概念群、1 セクション = 1 概念。逐次読みを前提とする。
